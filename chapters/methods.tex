
\chapter{Methods} \label{chap:methods}

In the following chapter the methods used in this thesis are presented. 
First the setup of curved \acrshort{tbl} simulations in \name{neko} is presented.
Then the generation of the \acrshort{zpg} conditions is described.
Finally, it is presented how the \acrshort{zpg} is imposed onto the \acrshort{dns} domain.

\section{Curved TBL simulations}

In the next step to derive the curved \acrshort{zpg} \acrshort{tbl} we first need to create a general setup for the curved \acrshort{tbl}. 
As a basis we use the setup of the plane \acrshort{tbl} and add a curved part at the end of the domain of the plane \acrshort{tbl}. 
The setup of the plane \acrshort{tbl} is described in \autoref{sec:planeTBL.}
The plane part is necessary such that the \acrshort{tbl} can develop properly since we want $Re_{\theta} \approx 1000$ when the curvature starts. 
The curved part has a constant radius and develops in negative wall-normal direction. 
Therefore, the \acrshort{tbl} experiences a convex curvature. 
At the end of the curvature another straight domain part is added which is called \benennung{\acrlong{so}}.
There is a smooth transition between the curved part and the \acrlong{so} because it follows the tangential direction of the end of the curved part. 
The complete set-up is shown in \autoref{fig:paraviewDomain}.
The \acrlong{so} is applied to reduce the impact of the \acrlong{bc} at the outflow. 
The pressure \acrshort{bc} as the outflow forces the pressure to be zero.
Therefore, the pressure trend in front of the outflow is impacted by that boundary condition.
To avoid this \acrshort{bc} impacts the pressure progression in the curvature, the \acrlong{so} is added.
The mesh is generated similar to the one used for the plane \acrshort{tbl} with the addition of the curved part and the \acrlong{so}.
The curved part is created with a circular arc with the corresponding values for the radius and angle.
In the \acrlong{so} the mesh is coarsened since this part of the domain is not relevant for the study of the curved \acrshort{tbl}. \\

\subsubsection{Boundary and initial conditions}
The imposed boundary and initial conditions are similar to the ones of the plane \acrshort{tbl} simulations.
The initial condition is as in the plane \acrshort{tbl} a Blasius profile.
The outflow condition and the boundary condition at the lower wall are transferred as well.
As boundary condition for the top wall a Dirichlet condition is applied.
Since the domain is curved the coordinate system needs to be adapted.
We introduce a moving coordinate system that is based on the streamwise direction $s$, wall-normal direction $n$ and the spanwise-direction $z$.
The corresponding velocities are named as $u$, $v$ and $w$.
Hence, the Dirichlet condition with the velocity components $(1,0,0)$ is applied in the new coordinate system.
In \autoref{sec:planeTBL} normal outflow and Dirichlet as boundary condition on the top wall are compared regarding their effect on the \acrshort{tbl} development.
These verifications show that the differences at last on a flat \acrshort{tbl} are negligible small, meaning that the Dirichlet boundary condition does not prevent the growth of the \acrshort{tbl}.
Therefore, we transferred this finding to the curved \acrshort{tbl}, which also brings some simplifications in the case set-up.
The application of the Dirichlet condition is possible in the f90 file, where the normal and tangential velocity components can be calculated and imposed as user velocity. 
The normal outflow condition is in neko implemented and until now only works for normal directions axis-aligned to a fix x-y-z coordinate system. 
As initial condition the Blasius boundary layer was used similar as for the plane \acrshort{tbl}.
In the curved part the Blasius profile is rotated such that it builds up in wall-normal direction. \\


This setup was used to create three different variations of curved \acrshort{tbl}.
We tested two different radii to study the impact of different curvatures. 
We used the radii of $R_1 = 100 \cdot \delta_{99}$ and $R_2 = 500 \cdot \delta_{99}$ with $\delta_{99}$ measured in the plane \acrshort{tbl} at $Re_{\theta} = 1000$. 
Two setups where tested with radius $R_1$ namely with $30$° and $90$°. 
For the second only one setup was tested with $18$° curvature. 
This value is chosen since it has the same arc length as the case with $R_1$ and $90$° curvature. 
Therefore, the cases have a comparable curved section length.
The domain dimensions for the tested cases are summarised in \autoref{tab:curved_cases}.

\section{Zero-pressure gradient generation}

In this section the procedure to generate a \acrshort{zpg} in the curved \acrshort{tbl} domain is introduced. 
This procedure is done on \name{OpenFOAM} using the Bayesian optimiser code developed by \citet{morita_applying_2022}. 
As described in \autoref{sec:OptimsationBasics} the code is able to optimize the shape of the domain to achieve a given Clauser parameter, which in this case is a ZPG. 
To do this the simulation setup has to be transferred to \name{OpenFOAM}. 
In the following paragraph the necessary steps are described in detail. 

\subsection{OpenFOAM setup}\label{subsec: methodsOFsetup}

\paragraph{General settings}
With \name{OpenFOAM} the \acrshort{rans} simulations for the optimisation are performed.
As described in \autoref{sec:SimBasics} \acrshort{rans} simulations use turbulence models compared to \acrshort{dns} where the turbulence is simulated completely.
There are different commonly used models to describe the turbulence which have various properties and accuracies.
In this simulation we use the so-called $k-\omega-sst$-model.
This model is a combination of the $k-\omega$- and $k-\varepsilon$ model with an added term for the shear-stress transport (SST) \citep{menter_two-equation_1994}.
in the sublayer of the boundary layer the $k-\omega$ model is used because of its accuracy in this region. 
This is blended with the $k-\varepsilon$ model which gives better results for $k$ and $\varepsilon$.
These quantities determine the eddy viscosity which is the main information for the solver of the mean flow. 
Therefore, a reasonable prediction of mean flow depends on accurate values for $k$ and $\varepsilon$ \citep{menter_two-equation_1994}.

\subsubsection{Mesh Generation}
The transfer of the curved TBL simulation setup to \name{OpenFOAM} includes some adaptations which will be presented here. 
First, the mesh has to be regenerated. 
It is not possible to use a mesh created in \name{gmsh} and transfer it to \name{OpenFOAM}, since the code has to adapt the domain and the mesh after every iteration of the optimization.
Therefore, the domain has to be regenerated in every iteration with \verb|blockMesh|.
As a consequence the domain has to be defined in the \verb|blockMeshDict|.
To ensure that the results from the \name{OpenFOAM} simulation are transferrable into \name{neko}, we have to ensure a quite similar setup.
Therefore the domain has the same curvature and also an inlet and outlet region similar to the constitution of the \acrshort{dns} domain.
However the inlet is shorter than the one in the \acrshort{dns} domain.
The initial condition for the \name{OpenFOAM} simulation is extracted at a position $s_{extr}$ in the straight section of the \acrshort{dns} domain.
Therefore the boundary layer in the \acrshort{rans} simulation only needs the remaining straight part $s_{straight} - s_{extr}$ to develop similar to the \acrshort{dns} boundary layer.
This is shown in \autoref{fig:openfoamDomain}.
Compared to the domain used for the DNS, the height of the domain is increased to $10 \cdot \delta_{99}$ for R100deg30 and R100deg90, and up to $20 \cdot \delta_{99}$ in R500deg18.
This modification brings some simplifications for transferring the results of the optimised \name{OpenFOAM} simulation to the DNS domain which will be described in \autoref{subsec:data_extr}.
To enable the adaption of the \benennung{design points}, the upper wall cannot be defined as fix circular arc in the curved section.
Instead it is defined as spline defined by the \benennung{design points}.
$x_{\theta_i}$, $y_{\theta_i}$ are the coordinates that describe the position of the \benennung{design points} $\theta_i$ whose position is modified to minimise the \benennung{objective function}.  
The optimisation code is constructed in a way that it modifies the position of the \benennung{design points} in wall-normal direction. 

\fxinfo{hier omrge weitermachen}
Since the \benennung{design space} is curved, the wall-normal direction does not correspond to a fixed axis. 
Hence, their position has to be transformed in $x-$ and $y-$ coordinates to correctly define the spline.
It has to be noticed that the straight inlet region is also part of the spline, due to the positioning of the \benennung{design points}.
This will be discussed in detail in \autoref{subsec:opt_settings}.
The lower wall is defined as a straight edge first and then transfers into a circular arc.
The tail is not part of the spline, since there no optimisation is needed.
It is meshed in a block with straight vertices.
The resolution of the mesh increases near the wall to ensure $y^+ = 1$ near the lower wall.

\paragraph{Initial and boundary conditions}
The quantities that need predefined initial conditions depend on the chosen RANS model (XXQuelle). 
Since in this project the $k-\omega-sst$-model is used, it is necessary to define initial conditions for turbulent kinetic energy $k$, dissipation $\omega$ and velocity $U$. 
This is done due to the imposition of an initial field at the inflow which is extracted from the previous DNS simulation of the curved TBL (see XX).
Therefore, a slice of the fully developed \acrshort{tbl} from the DNS simulation is taken.
We take this slice at the streamwise position in the plane part of the domain where $Re_{theta} = 1000$ is reached is identified.
At this streamwise position the fields of $U$, $k$ and $\omega$ are extracted along the height of the domain.
In order to do this we perform a postprocessing of the DNS simuation whith the \name{OpenFOAM}- mesh used as interpolation points.
With the \name{Python} package \benennung{fluidfoam} it is possible to read an \name{OpenFOAM} mesh. 
This is used to extract the points of the \name{OpenFOAM} mesh at its inflow.
These points are used as interpolation points to perform the postprocessing. 
As described earlier the domain used in \name{OpenFOAM} is higher than the one for the DNS simulations.
The usage of this higher domain would lead to interpolation conflicts since there are no data for the upper interpolation points. 
To avoid these conflicts the interpolation points lying beyond the height of the DNS domain are cutted.
They will be filled manually at the end of the script.
With these interpolation points the postprocessing script starts until one can extract the values.
The values for $U$ can be extracted directly, whereas $k$ and $\omega$ need to be computed. 
To use these fields in the higher \name{OpenFOAM} domain additional values for the upper part of the inflow are manually added. 
For this part of the inflow the last interpolated value is copied and added in the remaining cells.

\subsection{Data extraction} \label{subsec:data_extr}

Once the optimisation of the \name{OpenFOAM} simulation is finished, the next step is to transfer the relevant quantities into the \name{neko} simulation.
As stated before, the domain use for the DNS is smaller in wall-normal direction. 
Threfore the shape modification from the RANS simulation is not transferred.
However, only the qunatities influenced by the shpae adaptation are transferred.
Since the pressure itself is not direclty adapted the velocity components build the defining quantities.
Therefore the velocity components are extracted at the height where the DNS domain has its upper wall, such that they can be imposed as boundary condition.
The extraction of the data from the \name{OpenFOAM} simulation is done with \name{Python}.
First the simulation with the best optimization result is detected and stored in a separate directory.
The \name{Python} script reads the coordinates from the C/Cx/Cy files and the velocites from the corresponding fields. 
At a user-defined height the velocity values then are interpolated from the surrounding cells. 
The interpolated velocity are stored in a csv file. 
The integration of these values in the \acrshort{dns} is described in the next section.

THe setup of the simulations with the optimized boundary conditions at the upper wall have despite from this similar settings as the ones for the curved \acrshort{tbl} described in XX.

\paragraph{Velocity integration}

To run the \acrshort{dns} with \acrfull{zpg} the velocities extracted from \name{OpenFOAM} need to be applied as boundary condition on the \name{neko} domain. 
This is implemented in the user f90 file. 
There the boundary points are read from the mesh. 
Then the streamwise position s for the current point is calculated.
With this positon the mehtod interpolate velocity s is called.
In this method the velocity is linearly interpolated form the surrounding velocity values read from the csv file.
This is done for all streamwise positons that are included in the csv file. 
Hence, also a part of the straight inflow in front of the curvature, and a part of the tail are included.
In the parts of the domain that are not included, (1,0,0) is applied as before. 
To avoid sharp changes of the boundary condition when changing from 1,0, to the optimized velocities, a blending is inclued.
This blends the edge value of the optimized velocities into the 1,0,0 region in 10 length units in streamwise direciton.
