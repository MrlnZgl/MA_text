
\chapter{Methods and Setup} \label{chap:methods and Setup}



In this chapter, the methods employed in this thesis are provided. 
First, we present \acrfull{sem} and simulation method and outline the setup of curved \acrshort{tbl} simulations in \name{neko}.
Next the optimisation method to derive the \acrshort{zpg} conditions is described.
Finally, the methodology employed to impose the \acrshort{zpg} onto the \acrshort{dns} domain is presented.

\section{Spectral element method}

\section{Curved turbulent boundary layer simulation setup}\label{sec:MethodCurvedTBL}

In order to derive the curved \acrshort{zpg} \acrshort{tbl}, we first need to create a general setup for the curved \acrshort{tbl}. 
The initial configuration of the plane  \acrshort{tbl} is utilised as the basis, and we add a curved part at the end of the domain of the plane \acrshort{tbl}. 
The setup of the plane \acrshort{tbl} is described in \autoref{sec:planeTBL}.
It is imperative to consider the plane part, as this is necessary for the optimal development of the \acrshort{tbl}. The objective is to achieve
a value of $Re_{\theta} \approx 1000$ when the curvature starts. 
The curved part has a constant radius and develops in negative wall-normal direction. 
Therefore, the \acrshort{tbl} experiences a convex curvature. 
At the end of the curvature, an additional straight domain segment is appended, which is called \benennung{\acrlong{so}}.
A seamless transition is observed between the curved segment and the straight one \acrlong{so}, as the former aligns with the tangential direction of the curved segment. 

The complete setup is shown in \autoref{fig:paraviewDomain}.
The \acrlong{so} is introduced to reduce the impact of the \acrlong{bc} at the outflow. 
The pressure \acrshort{bc} at the outflow forces the pressure to be zero.
Consequently, the pressure trend in front of the outflow is affected by the specified boundary condition.
In order to circumvent  that the \acrshort{bc} impacts the pressure progression in the curvature, the \acrlong{so} is added.
The mesh is generated in a manner analogous to that employed for the plane  \acrshort{tbl} with the addition of the curved part and the \acrlong{so}.
The curved segment is created with a circular arc with the corresponding values for the radius and angle.
In the \acrlong{so} the mesh is coarsened since this part of the domain is not relevant for the study of the curved \acrshort{tbl}. \\

\subsubsection{Boundary and initial conditions}
The imposed boundary and initial conditions are similar to those of the plane \acrshort{tbl} simulations.
The initial condition is a \benennung{Blasius profile}, as in the plane \acrshort{tbl} simulations (see \autoref{sec:planeTBL}).

\comment{hier blasius profile beschreiben}
The Blasius profile is an analytic solution for the velocity profile of a laminar boundary layer along a flat plate with zero pressure gradient \citep{blasius_grenzschichtern_1908}.
It is based on the self-similar character of the boundary layer meaning the velocity profile along the flat plate looks similar at any streamwise position once it is rescaled with the local boundary layer thickness \citep{schlichting_boundary-layer_2017}.
The outflow condition and the lower wall boundary condition are also transferred.
A \benennung{Dirichlet condition} is applied as boundary condition for the top wall.
Since the domain is curved the coordinate system needs to be adapted.
We introduce a moving coordinate system that is based on the streamwise direction $s$, the wall-normal direction $n$ and the spanwise direction $z$.
The corresponding velocities are named as $u$, $v$ and $w$.
Hence, the Dirichlet condition with the velocity components $(1,0,0)$ is applied in the new coordinate system.
In \autoref{sec:planeTBL}, normal outflow and the Dirichlet  boundary condition on the top wall are compared  in terms of their effect on the \acrshort{tbl} development.
These validations show that, ultimately, the differences on a flat \acrshort{tbl} are negligible, meaning that the Dirichlet boundary condition does not prevent the growth of the \acrshort{tbl}.
Therefore, we transferred this finding to the curved \acrshort{tbl}, which also simplifies the scenario setup.
The Dirichlet condition can be applied in the \texttt{f90} file, where the normal and tangential velocity components can be calculated and set as the user velocity. 
The normal outflow condition has been  implemented in neko, but currently only works for normal directions axis-aligned to a fixed $xyz$-coordinate system. 
The Blasius boundary layer was used as the initial condition similar to the plane \acrshort{tbl}.
In the curved part the Blasius profile is rotated so that it builds up in the wall-normal direction. \\

\section{Optimisation method}

This section introduces the procedure for generating a \acrshort{zpg} in the curved \acrshort{tbl} domain. 
This procedure is performed using \name{OpenFOAM} and the Bayesian optimiser code developed by \citet{morita_applying_2022}. 
As described in \autoref{sec:OptimsationBasics}, the code is able to optimise the shape of the domain to achieve a given Clauser parameter,  in this case the parameter is a ZPG. 
To achieve this, the simulation setup has to be transferred to \name{OpenFOAM}. 
In the following paragraph the necessary steps are described in detail. 

\subsection{Allg über OPenFOAM funktionsweise}

\subsection{OpenFOAM simulation setup}\label{subsec: methodsOFsetup}

The OpenFOAM software is used to perform the RANS simulations for optimisation.
As described in \autoref{sec:SimBasics} \acrshort{rans} simulations use turbulence models, whereas \acrshort{dns} simulate the turbulence completely.
There are various commonly used models for describing the turbulence, which have different properties and accuracies.
In this simulation we use the so-called $k\text{-}\omega$ SST model.
This model combines the $k\text{-}\omega$- and $k\text{-}\varepsilon$ models, incorporating an additional term for the shear-stress transport (SST) \citep{menter_two-equation_1994}.
In the sublayer of the boundary layer, the $k\text{-}\omega$ model is used because of its accuracy in this region. 
This is blended with the $k\text{-}\varepsilon$ model, which provides more accurate results for $k$ and $\varepsilon$.
These quantities determine the eddy viscosity which is the main information required by the mean flow solver. 
Therefore, an accurate prediction of mean flow depends on accurate values for $k$ and $\varepsilon$ \citep{menter_two-equation_1994}. \\

\subsubsection{Mesh generation}
Transferring the curved TBL simulation setup to \name{OpenFOAM} requires some adaptations which will be presented here. 
First, the mesh has to be regenerated. 
It is not possible to use a mesh created in \name{gmsh} and transfer it to \name{OpenFOAM}, since the code must adapt the domain and the mesh after each iteration of the optimisation.
Therefore, the domain has to be regenerated with \verb|blockMesh| in every iteration.
As a consequence, the domain has to be defined in the \verb|blockMeshDict|.
To ensure that the results from the \name{OpenFOAM} simulation can be transferred into \name{neko}, the setup must be quite similar.
Therefore the domain has the same curvature as well as  inlet and outlet regions similar to the constitution of the \acrshort{dns} domain.
However, the inlet is shorter than the one in the \acrshort{dns} domain.
The initial condition for the \name{OpenFOAM} simulation is extracted at a position $s_{extr} = 275$ in the straight section of the \acrshort{dns} domain.
Therefore, the boundary layer in the \acrshort{rans} simulation only needs a much shorter straight inlet region to develop similar to the \acrshort{dns} boundary layer.
We defined the remaining length of the straight inlet as the remaining part of $s_{i} - s_{extr}$ and then added a balance zone of $s_{bal} = 20$ which ends up in a straight inlet of $s_{iOF} = 70$.
This balance region is recommended since quantities of the extracted \acrshort{dns} inflow profile have to be rebalanced in the \acrshort{rans} model (XXQUELLE).
In \autoref{fig:openfoamDomain} the general configuration of the \acrshort{rans} domain is depicted exemplary for the R100deg30 case.
Compared to the domain used for the DNS, the height of the domain is increased to $10 \cdot \delta_{99}$ for R100deg30 and R100deg90, and up to $20 \cdot \delta_{99}$ in R500deg18.
This modification simplifies the transfer of the results of the optimised \name{OpenFOAM} simulation to the DNS domain, as  described in \autoref{subsec:data_extr}.
To enable the adaption of the \benennung{design points}, the upper wall cannot be defined as a fixed circular arc in the curved section.
Instead, it is defined as a spline defined by the \benennung{design points}.
The coordinates $x_{\theta_i}$ and $y_{\theta_i}$  describe the position of the \benennung{design points} $\theta_i$ which are modified to minimise the \benennung{objective function}.  
The optimisation code is constructed in a way that it modifies the position of the \benennung{design points} in the wall-normal direction. 


Since the domain is curved, the wall-normal direction does not correspond to a fixed axis. 
Therefore, the \benennung{design points} $\theta_i$ must be transformed into  $x-$ and $y-$ coordinates to define the spline correctly.
This will be discussed in detail in \autoref{subsec:opt_settings}.
The lower wall is defined first as a straight edge and then transitions into a circular arc.
The straight outlet is not part of the spline, since there no optimisation is required.
It is meshed as a block with straight vertices.
The mesh resolution increases near the wall to ensure $y^+ = 1$ near the lower wall.

\subsubsection{Boundary and initial conditions}
The quantities that require predefined initial conditions depend on the selected RANS model (XXQuelle). 
Since  the $k-\omega-sst$-model is used in this project, it is necessary to define the initial conditions for turbulent kinetic energy $k$, dissipation $\omega$ and velocity $U$. 
This is achieved by applying an initial field at the inflow which is extracted from the previous \acrshort{dns} simulation of the curved \acrshort{tbl}.
Therefore, a slice of the fully developed \acrshort{tbl} from the DNS simulation is used.
We take this slice at the streamwise position in the straight inlet where $Re_{theta} = 1000$ is reached.
At this streamwise position the fields of $U$, $k$ and $\omega$ are extracted across the entire height of the domain. 
In the spanwise direction the fields are already averaged which corresponds to the two-dimensionality of the \name{OpenFOAM} simulation.
In order to do this we perform a post-processing of the \acrshort{dns} simulation using the \name{OpenFOAM} mesh as interpolation points.
With the \name{Python} package \texttt{fluidfoam} it is possible to read the points from an \name{OpenFOAM} mesh into \name{Python}. 
This is used to extract the points of the \name{OpenFOAM} mesh at its inflow.
These points are used as interpolation points to perform the post-processing. 
As described earlier the domain used in \name{OpenFOAM} is higher than the one for the \acrshort{dns} simulations.
The usage of this higher domain would lead to interpolation conflicts since there are no data for the upper interpolation points. 
To avoid these conflicts the interpolation points lying beyond the height of the \acrshort{dns} domain are cut.
They will be filled manually at the end of the script.
The values for $U$ can be extracted directly, whereas $k$ and $\omega$ need to be computed. 
To use these fields in the higher \name{OpenFOAM} domain additional values for the upper part of the inflow are manually added. 
For this part of the inflow the last interpolated value is copied and added in the remaining cells.

\subsubsection{Optimisation algorithm}

The optimisation algorithm is based on a Bayesian optimisation with \acrfull{gpr}.
\fxinfo{hier noch merh zur optimierer theorie}
The algorithm was first introduced by \citet{morita_applying_2022} and tested for different cases by \citet{ganagdhar_bayesian_2025}.
The workflow of the algorithm is depicted in \autoref{fig:optiFlowchart} and can be split into 6 steps.
First, the next samples for $\theta_i$ are proposed by the \acrshort{gpr}.
The first iteration are randomly taken samples. 
Then, the \acrshort{gpr} suggests the next combination based on all previous iterations.
They are transferred into the samples into $x_{\theta_i}$ and $y_{\theta_i}$ to include them into the \texttt{blockMeshDict}.
In the next step the \name{OpenFOAM} simulation starts which consists of three steps.
First the new mesh is generated, then the case is split to the processors and then the simulation is run.
Once the \name{OpenFOAM} simulations ends, multiple boundary layer quantities are calculated to determine the Clauser parameter. 
With this, the residual, which is defined by the normalised $L^2$-norm of $\beta(s)$, is calculated as
\begin{equation}
    R = \frac{\lVert \beta_{\text{eval}} - \beta_{\text{target}} \rVert_2}{\sqrt{N_{\text{eval}}}}
    \label{eq:residual}
\end{equation}

with $\beta_{eval}$ representing the calculated value in the current iteration.
$\beta_{target}$ is the objective value. 
The $L^2$-norm is normalised by the number of cells that are included in the evaluation window $N_eval$.
That is necessary for comparisons between residuals of different cases and respectively different evaluation window sizes.
In the next step the setup of the \texttt{OFcase} is stored if the residual is below a user-defined threshold.
That is good to later compare different configurations with comparable good residuals.
In the last step the \benennung{design points} are added to the \texttt{gpList.dat} and the convergence check is performed.
These steps are repeated until the residual is converged or the maximum number of iterations is reached.

\subsubsection{Optimisation parameters}

Besides the positioning of the \benennung{design points} other parameters have to be determined for the optimisation algorithm.
For simplicity, all user-defined parameters are defined in a single \texttt{params.py} file.
There e.g. specific dimensions for the geometry of the domain can be defined, such as the radius, the length of the straight inlet and outlet and the angle.
The \texttt{blockMeshDict} such that the general configuration of the domain with straight inlet, curved section and straight outlet is predefined.
In addition, the position of the \benennung{control points} in streamwise direction can be defined.
Regarding the \benennung{design points}, the design space also has to be limited for all \benennung{control points}.
We adapt their \benennung{design space} iteratively starting with the same wide range for all \benennung{control points}.
During the optimisation process we observe the motion of each control points in its range.
If \benennung{control points} tend over multiple iterations to a limit of their \benennung{design space}, we interrupt the optimisaiton and adapt their range correspondingly.
To avoid a steady extension of the \benennung{design space}, the opposite limit is also changed to keep the \benennung{design space} size constant.
The optimisation then can be restarted, and the old optimisation results are still taken into account from the \acrshort{bo}.
However, these adjustments require caution as hasty modifications can distort the exploration of the \acrshort{bo}.
Therefore, these adjustments should only be made if there is a long-term trend.
Concerning the optimisation algorithm, the evaluation window at the lower wall has to be defined.
As visualised in \autoref{fig:OF_designPOints} the evaluation window does not cover the complete bottom wall of the domain.
Instead the evaluation window is confined to a section on which the modification of the \benennung{design points} has the capacity to exert an effect on $\beta(s)$ at the lower wall.
The most important parameter is the definition of the \benennung{objective function} for $\beta(s)$.

\subsection{Data extraction} \label{subsec:data_extr}

Once the optimisation of the Clauser parameter in \name{OpenFOAM} finished, the next step is to transfer the relevant quantities into the \name{neko} simulation.
As stated before, the domain use for the \acrshort{dns} is less high in wall-normal direction. 
Therefore the shape modification from the \acrshort{rans} simulation is not transferred.
However, only the quantities influenced by the shape adaptation are transferred.
Since the pressure itself is not directly adapted the velocity components build the defining quantities.
The velocity components are extracted at the height where the \acrshort{dns} domain has its upper wall, such that they can be imposed as boundary condition.
The extraction of the data from the \name{OpenFOAM} simulation is done with \name{Python}.
First the simulation with the best optimisation result is detected and stored in a separate directory.
The \name{Python} script reads the coordinates from the \texttt{C/Cx/Cy} files and the velocities from the corresponding fields. 
At a user-defined height the velocity values then are interpolated from the surrounding cells. 
The interpolated velocity are stored in a \texttt{csv} file. 
The integration of these values in the \acrshort{dns} is described in the next section.

\paragraph{Velocity integration}

To run the \acrshort{dns} with \acrfull{zpg} the velocities extracted from \name{OpenFOAM} need to be applied as boundary condition on the \name{neko} domain. 
This is implemented in the \texttt{user.f90} file. 
There the boundary points are read from the mesh. 
Then the streamwise position $s$ for the current point is calculated.
With this position the method \texttt{interpolate_velocity(s)} is called.
In this method the velocity is linearly interpolated form the surrounding velocity values read from the \texttt{csv} file.
This is done for all streamwise positions that are included in the \texttt{csv} file. 
Hence, also a part of the straight inflow in front of the curvature, and a part of the tail are included.
In the parts of the domain that are not included, $(1,0,0)$ is applied as before. 
To avoid sharp changes of the boundary condition when changing from $(1,0,0)$ to the optimised velocities, a blending is included.
This blends the edge value of the optimised velocities into the $(1,0,0)$ region in 10 length units in streamwise direction.
