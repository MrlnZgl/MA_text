\section{Channel simulations}
In the beginning of the project channel simulations were performed. 
To get familiar with the \acrshort{dns} code and the effect of different settings these reproducable simulations were performed.
In addition, the created postprocessing script was verified with these results and reference results.
In the \name{neko} installation some examples are included which also contains one for a tubulent chnannel simulation.
We used this example as starting point and made some modifications.
The simulation is setup for a tubulent channel flow at $Re_\tau = 180$.
With the bulk velocity $U_b = 1$ and the channle ahlf height $\delta = 1$, the bulk Reynolds number is set to be $Re_b = 2800$.
The simulation runs with polynomial order 7 and time order 3.
At the top and the bottom wall we set \benennung{no-slip} as boundary condition.
The streamwise and spanwise wall are set to be periodic, meaning that everything that goes out at one side enters the domain at the opposite.
As a first step we refined the mesh which was necessary to create proper statistics.
For an adequate resolution we defined the grid spacing in plus units to be $\Delta x^+= 12$, $\Delta z^+= 6$. 
In wall-normal direction we set the point closest to the wall to be at $y^+ = 1$ and then coarsening the resolution.
The physical domain size is $(4\pi \delta, 2 \delta, \frac{4}{3}\pi \delta)$.
Then we applied some new initial conditions to create the turbulence.
We added more pertubations in different scales such that the transition into a fully turbulent flow is much faster.
In addition, the new pertubations made the old random pertubaitions in wall-normal direction become unnecessary (XX warum ist das gut??).
The new setup of inital conditons is depicted in \autoref{fig:channel_IC}.
With this setup simulations were performed.
The collected statistics are discussed in chapter XX.

\section{plane TBL simulations}

In the following part of the project we created a setup for a simulation of a \acrshort{tbl} acorss a plane wall.
These simulations are used to verify the general settings of a \acrshort{tbl} simulations, and to verify the results through the comparison with similar simulations.
The domain is defined as a box of the dimension $325 \times 24.5 \times 24.5$ which corresponds to $50\delta_{99} \times 4\delta_{99} \times 4\delta_{99}$ measured at a reference point at $Re_{\theta} = 1000$.
The resolution in streamwise and spanwise diriction is constant with $x^+= 12$ and $z^+ = 6$.
In wall-normal direction the resolution decreases with increasing distance from the lower wall starting at the wall with a resolution of $y^+ = 0.8$. 
In the middle the resolution is constant on aquite coarse level.
At the upper wall there is a small area where the resolution again increases since there the shear stress is measured (XXstimmt das??).
The mesh is created with \name{gmsh}. 
First a 2D mesh is constructed in the $x-y$/ streamwise- wall-normal-plane is created. 
With the mesh transform tool in \name{neko} it is then extended in the spanwise direction. 

\subsubsection{Boundary and initial conditions}

To setup the \acrlong{tbl} boundary and initla conditions are necessary. 
In the initial condition a boundary layer is imposed at the lower wall. 
Since a laminar boundary layer has the quite easy blasuis profile as solution, a laminar boundary layer is given as initial velocity condition. 
The velocity in the free-stream $U_0$ is set to be 1 initially.
AS boundary conditions we impose a no-slip condition on the lower wall.
The outflow has the boundary condition 'outlfow+dong', which means XXX.
For the upper wall we tested two different setups.
For one simulation we put a 'normal outflow' condition at the top wall, which allows outflow in normal direction across the domain end.
In the otehr simulation we imposed a Dirichlet condition with (1,0,0) as velocity profile. 
This is done to compare the effect of the different boundary conditions on the flow behaviour. 
This becomes especially importnat for the simulation of the curved \acrshort{tbl}, since there the boundary condition needs to be imposed piecewise on the top wall.

\paragraph{General settings}
With \name{OpenFOAM} the \acrshort{rans} simulations for the optimization are performed.
As described in chapter XX, \acrshort{rans} simulations use turbulence models compared to \acrshort{dns}, where the turbulenc is simulated completely.
There are different common used models to describe the turbulence which have various properties and accuracies.
In this simulaiton we use the so-called $k-\omega-sst$-model.
This model 
XX something about this model is a combinatio of the $k-\omega$- and $k-\epsilon$ model with an added term for the shear-stress transport (SST) \citep{menter_two-equation_1994}.
in the sublayer of the boundary layer the $k-\omega$ model is used because of its accuracy in this region. 
This is blende with the $k-\epsilon$ model which gives better results for $k$ and $\epsilon$.
These qunatities determine the eddy viscosity which is the main information for the solver of the mean flow. 
THerefore, a reasonable prediction of mean flow depends on accurate values for $k$ and $\epsilon$ \citep{menter_two-equation_1994}.

\paragraph{Mesh Generation}
XX pLus units
The transfer of the curved TBL simulation setup to \name{OpenFOAM} includes some adaptations which will be presented here. 
First, the mesh has to be regenerated. 
It is not possible to use a mesh created in \name{gmsh} and transfer it to \name{OpenFOAM}, since the code has to adapt the domain and the mesh after every iteration of the optimization.
Therefore, the domain has to be regenerated in every iteration with \verb|blockMesh|.
As a consequence the domain has to be defined in \verb|blockMeshDict|.
To ensure that the results from the \name{OpenFOAM} simulation is transferrable into \name{neko}, we have to ensure a quite similar setup.
Therefore the domain has the same curvature and also an inlet and outlet region similar to the \acrshort{dns} domain.
However the inlet is shorter than the one for the \acrshort{dns}.
The initial condition for the \name{OpenFOAM} simulaiton is extracted at a positon $s_{extr}$ in the inlet region of the \acrshort{dns} domain.
Therefore the boundary layer in the \acrshort{rans} simulation only needs the remaining inlet region $s_{straight} - s_{extr}$ to develop similar to the \acrshort{dns} boundary layer.
This is shown in \autoref{fig:openfoamDomain}.
Compared to the domain used for the DNS, the height of the domain is increased to $\delta_{99}*XX = XX$.
This modification brings some simplifications when the results of the optimized \name{OpenFOAM} simulation are transferred to the DNS domain which will be described later (see XX).
To enable the adaption of the \benennung{design coordinates} cannot be defined as fix circular arc.
Instead it is defined as spline.
The points defining the spline are the coordinates $x_{\theta_i}$, $y_{\theta_i}$ of the \benennung{design parameters} $\theta_i$ whose position is modified to minimize the \benennung{objective function}.  
The optimization code is constructed such that it modifies position of the \benennung{design parameters} in wall-normal direction. 
Since the \benennung{design space} is curved, the wall-normal direction does not correspond to a fixed axis. 
Hence, their position has to be transformed in $x-$ and $y-$ coordinates to define the spline correctly in space.
It has to be noticed that the straight inlet region is also part of the spline, due to the positioning of the \benennung{design parameters}.
This will be discussed in detail in XX.
To match the lower wall to this configuration we defined it also as spline but increased the number on control points to a very high level, such that it is quite similar to an analytical circular arc.
The tail is not part of the splines, since their no optimization is needed.
It is meshed in a block with straight vertices.
The resolution of the mesh increases near the wall to ensure $y^+ = 1$ near the lower wall.

\paragraph{Initial conditions or boundary condition ??XX}
The quantities that need predefined initial conditions depend on the chosen RANS model (XXQuelle). 
Since in this project the $k-\omega-sst$-model is used, it is necessary to define initial conditions for turbulent kinetic energy $k$, dissipation $\omega$ and velocity $U$. 
This is done due to the imposition of an initial field at the inflow which is extracted from the previous DNS simulation of the curved TBL (see XX).
Therefore, a slice of the fully developed \acrshort{tbl} from the DNS simulation is taken.
We take this slice at the streamwise position in the plane part of the domain where $Re_{theta} = 1000$ is reached is identified.
At this streamwise position the fields of $U$, $k$ and $\omega$ are extracted along the height of the domain.
In order to do this we perform a postprocessing of the DNS simuation whith the \name{OpenFOAM}- mesh used as interpolation points.
With the \name{Python} package \benennung{fluidfoam} it is possible to read an \name{OpenFOAM} mesh. 
This is used to extract the points of the \name{OpenFOAM} mesh at its inflow.
These points are used as interpolation points to perform the postprocessing. 
As described earlier the domain used in \name{OpenFOAM} is higher than the one for the DNS simulations.
The usage of this higher domain would lead to interpolation conflicts since there are no data for the upper interpolation points. 
To avoid these conflicts the interpolation points lying beyond the height of the DNS domain are cutted.
They will be filled manually at the end of the script.
With these interpolation points the postprocessing script starts until one can extract the values.
The values for $U$ can be extracted directly, whereas $k$ and $\omega$ need to be computed. 
To use these fields in the higher \name{OpenFOAM} domain additional values for the upper part of the inflow are manually added. 
For this part of the inflow the last interpolated value is copied and added in the remaining cells.

\subsection{Data extraction} \label{subsec:data_extr}

Once the optimisation of the \name{OpenFOAM} simulation is finished, the next step is to transfer the relevant quantities into the \name{neko} simulation.
As stated before, the domain use for the DNS is smaller in wall-normal direction. 
Threfore the shape modification from the RANS simulation is not transferred.
However, only the qunatities influenced by the shpae adaptation are transferred.
Since the pressure itself is not direclty adapted the velocity components build the defining quantities.
Therefore the velocity components are extracted at the height where the DNS domain has its upper wall, such that they can be imposed as boundary condition.
The extraction of the data from the \name{OpenFOAM} simulation is done with \name{Python}.
First the simulation with the best optimization result is detected and stored in a separate directory.
The \name{Python} script reads the coordinates from the C/Cx/Cy files and the velocites from the corresponding fields. 
At a user-defined height the velocity values then are interpolated from the surrounding cells. 
The interpolated velocity are stored in a csv file. 
The integration of these values in the \acrshort{dns} is described in the next section.

\paragraph{Velocity integration}

To run the \acrshort{dns} with \acrfull{zpg} the velocities extracted from \name{OpenFOAM} need to be applied as boundary condition on the \name{neko} domain. 
This is implemented in the user f90 file. 
There the boundary points are read from the mesh. 
Then the streamwise position s for the current point is calculated.
With this positon the mehtod interpolate velocity s is called.
In this method the velocity is linearly interpolated form the surrounding velocity values read from the csv file.
This is done for all streamwise positons that are included in the csv file. 
Hence, also a part of the straight inflow in front of the curvature, and a part of the tail are included.
In the parts of the domain that are not included, (1,0,0) is applied as before. 
To avoid sharp changes of the boundary condition when changing from 1,0, to the optimized velocities, a blending is inclued.
This blends the edge value of the optimized velocities into the 1,0,0 region in 10 length units in streamwise direciton.
